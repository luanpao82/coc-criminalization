Stage 1 · Automated PDF → comprehensive raw data file

Goal: produce a single data file that mirrors the manually-coded coc_apps_all_info.xlsx but extends coverage from FY2024-only (hand-coded by the RA team) to all three years, and captures every HUD CoC Application field as a column — even the ones this study does not yet use.

Output: coc_raw_data.xlsx (677 rows × 348 columns; ~2 MB).

  • Row universe: one row per source document in file_inventory.csv — every PDF or DOCX that a CoC submitted across FY2022, FY2023, and FY2024.
  • Column schema: the 331 canonical HUD variables from codebook.md, in PDF question order (1A → 1B → 1C → 1D → 1E → 2A → 2B → 2C → 3A → 3B → 4A), plus 9 meta columns and 8 completeness/PLE-status columns.
  • Extractor: format-aware adapters (extract_pdf_native.py, extract_docx.py) with an anchor-then-parse rule — every question is located by its 1B-1./1D-4.-style anchor, and only the local block is parsed. A failure in one block cannot corrupt neighboring variables. Categorical outputs are validated against the codebook's controlled vocabularies; out-of-vocab values are flagged rather than coerced.
  • Cross-year harmonization: FY2024 field IDs are canonical. FY2022/23 content is crosswalked into the same columns where the construct is equivalent (see crosswalk.csv and panel_field_map.csv).

Known gaps (surfaced by the Stage 1 extractor):

  • ~20 scanned PDFs — anchor text not present in the text layer. OCR (ocrmypdf) infrastructure is in place but the task has not been run.
  • 19 Special NOFO documents — structurally different HUD form (unsheltered funding cycle, FY2022). These do not contain the standard 1D section and are flagged accordingly.
  • 3 mis-filed CoC Program Project Applications (SF-424) instead of the Consolidated Application. Require re-sourcing from HUD.

Stage 2 · Text extraction for research-used variables

Goal: populate free-text narrative cells for the variables this study measures. Doing Stage-2 narrative coding for all 24 narrative variables in the codebook is a large undertaking; we scoped text extraction to the three People with Lived Experience (PLE) engagement questions because they are the study's primary IV.

Fields extracted:

  • 1d_10 — Involving PLE in Service Delivery and Decisionmaking (FY2024 anchor; FY2022/23 equivalent at 1d_11, crosswalked in)
  • 1d_10b — Professional Development and Employment Opportunities for PLE
  • 1d_10c — Routinely Gathering Feedback and Addressing Challenges of PLE

Method: year-branched regex anchors (anchors_by_year in extract_narrative.py) with a 1E-1 backstop that guarantees every slice is bounded. Every extracted text cell carries a *_source_page and *_status annotation so reviewers can cross-reference the original PDF.

Stage 3 · Narrative coding into structured variables

Goal: turn each PLE narrative into a small set of machine-readable atomic codes that can enter a regression, while keeping every code defensible against its source text.

Output: coc_raw_for_coding.xlsx (677 rows × 41 columns; ~1 MB). One sheet, one row per CoC-year.

The coding protocol

  1. Atomic-code schema — 15 boolean indicators + one free-form frequency string across the 3 fields. Codes are grounded in Arnstein's (1969) participation ladder: representation (ladder top), compensation (tokenism defense), feedback loop (delegated power).
    • ple_umbrella (6): on_board, on_committees, compensated, hiring_advertised, formal_policy, decisionmaking
    • ple_prof_dev (5): paid_positions, comp_policy_formal, training_pipeline, career_advancement, scope_beyond_tokenism
    • ple_feedback (4 + freq): mechanism_formal, acts_on_feedback, addresses_barriers, closes_the_loop, frequency
  2. Claude Code reads each narrative and returns a JSON object with every code plus a one-sentence summary and a list of evidence: up to three verbatim quotations from the source text that justify the coded values. If a field is not discussed in the narrative, the code is null; the model is instructed never to fabricate.
  3. Scoring rule (Krippendorff 2018 §4.3, direct inference): True → 1, False → 0, null → 0. Rationale: HUD's instrument directly elicits the practice, so a CoC's non-mention is substantive evidence of absence, not of uncertainty. A robustness column pair (*_availcase) computes the available-case mean that skips nulls, for sensitivity checks.
  4. Composite indices are computed per CoC-year (range 0 → 1):
    • ple_representation = mean(on_board, on_committees, formal_policy, decisionmaking)
    • ple_compensation = mean(compensated, paid_positions, comp_policy_formal)
    • ple_feedback_loop = mean(mechanism_formal, acts_on_feedback, addresses_barriers, closes_the_loop)
    • ple_institution = mean of the three sub-indices — the headline IV

Coauthor review requested

0 CoC-years are currently fully coded (the pilot batch). Before scaling coding to the remaining ~600 CoC-years, we need coauthor sign-off on three things:

  1. Evidence ↔ code fidelity. Open coc_raw_for_coding.xlsx, sort by coc_id and year, and for each coded row check that the quotations in the three *_evidence columns actually support the 0/1 values in the atomic-code columns. If any code is not supported by its evidence, flip it in the cell and note the issue in reviewer_notes.
  2. Schema validity. Are the 15 atomic indicators the right operationalization of institutionalized PLE engagement? Should any be added (e.g., youth-specific representation) or dropped (e.g., hiring_advertised if too weak)? Composite-index formulas are reported above and can be adjusted.
  3. Scoring convention. We score null → 0 (absent mention = absent practice). If coauthors prefer the available-case convention (skip nulls entirely) as the primary, say so — both columns already exist in pilot_ple_variables.csv.

Once items 2 and 3 are confirmed, the pilot's 18 × 3 = 54 narrative codings scale to ~1,800 across the full corpus. Iterating the schema after full coding is expensive; now is the right moment to weigh in.

Stage 4 · Analytical panel for the regression

Goal: select from the comprehensive raw file only the variables the primary multilevel DiD specification needs, and join in the PLE composite indices. This is the file the analysis scripts read.

Output: coc_panel_wide_with_ple.xlsx (651 CoC-years × 264 columns). Built by merge_ple_variables.py from the existing coc_panel_wide.xlsx plus drafts/pilot_ple_variables.csv.

  • Row scope: 651 = 172 (FY2022) + 195 (FY2023) + 284 (FY2024), after excluding Special-NOFO submissions that do not carry the DV. Balanced subset (125 CoCs observed in all three years) is a sheet within coc_analysis_ready.xlsx.
  • Carried-in variables from coc_panel_wide.xlsx: all 245 structured variables used for the DV construction, political geography, organizational controls, and the Mundlak within-CoC means.
  • New PLE columns (from pilot_ple_variables.csv): the four composite indices plus 15 atomic bool indicators. Currently non-null for the 0 pilot-coded CoC-years; NaN elsewhere until Stage 3 coding is expanded.
  • Used by: run_multilevel.py (primary DiD), run_balanced_sensitivity.py, run_dv_robustness.py. The existing specifications run unchanged; PLE variables enter as new regressors in the extended spec.
